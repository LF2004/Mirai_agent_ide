4. 核心复杂框架设计
4.1 Agent 调度循环（稳定可靠）
借鉴 Cursor 的 Agent 交互模式，实现一个 状态机驱动的调度循环，避免死循环和卡死：

text
状态机:
  IDLE → 收到用户消息 → PROCESSING
  PROCESSING → 调用 LLM → 解析响应
  → 若需工具调用 → TOOL_CALLING → 执行工具 → 结果回传 → PROCESSING
  → 若无需工具 → TURN_ENDED → 回送结束帧 → IDLE
  任何步骤超时/异常 → ERROR → 回送错误帧 → IDLE

关键优化:
- 每个步骤设置超时 (如 LLM 调用 60s, 工具执行 30s)
- 循环计数器：连续工具调用超过 10 轮强制终止
- 熔断器：同一会话连续出错 3 次进入降级模式 (仅透传)
- 看门狗：主进程定时检测子进程状态，异常时自动重启
4.2 工具执行引擎 (ToolEngine)
设计原则：可扩展、安全、可观测。

typescript
// 工具定义接口
interface ToolDefinition {
  name: string;                // 如 'read', 'search_web'
  description: string;
  parameters: JSONSchema;
  execute: (args: any, context: ToolContext) => Promise<ToolResult>;
  timeout?: number;
  allowedPaths?: string[];     // 限制文件操作范围
}

// 内置工具注册
ToolEngine.registerBuiltinTools({
  read: { ... },
  grep: { ... },
  shell: { ... },
  // ...
});

// 用户自定义工具 (通过配置文件或 UI 添加)
ToolEngine.registerUserTool({
  name: 'search_web',
  description: 'Search the web for current information',
  parameters: { query: { type: 'string' } },
  execute: async ({ query }) => {
    // 调用外部搜索 API
    return await fetchSearchResults(query);
  }
});
工具调度策略：

串行：按顺序执行 (保证依赖)

并行：无依赖的工具可并行 (提升效率)

超时控制：每个工具执行带独立超时

重试机制：可重试的错误 (如网络超时) 自动重试一次

4.3 长任务/多任务管理 (Harness 风格)
实现类似 Harness 的 Task/Workflow 抽象，支持复杂任务拆解与汇总：

typescript
// 任务定义
interface Task {
  id: string;
  type: 'plan' | 'execute' | 'verify' | 'ask';
  status: 'pending' | 'running' | 'completed' | 'failed';
  dependencies?: string[];    // 依赖的任务 ID
  payload: any;              // 任务具体数据
  result?: any;
  maxRetries?: number;
  retryCount?: number;
}

// 多任务管理器
class TaskManager {
  private tasks: Map<string, Task>;
  private sessionId: string;

  // 根据 Plan 模式生成的计划拆解成子任务
  createTasksFromPlan(plan: Plan): Task[];

  // 调度执行：拓扑排序，并行执行无依赖任务
  executeTasks(tasks: Task[]): AsyncGenerator<TaskResult>;

  // 任务完成时汇总结果，决定下一步
  onTaskComplete(task: Task): void;
}
状态持久化：关键任务状态写入 SQLite，即使 IDE 重启也能恢复未完成的任务。

4.4 提示词管理 (PromptManager)
分层提示词架构：

技能层：加载 skills/ 目录下预定义的技能提示词 (如 React 最佳实践、TDD 流程)

动态注入时机：每次构造上游请求前，根据当前会话状态组装完整提示词。

6. 安全性设计
6.1 进程隔离
渲染进程无法直接访问 Node API，所有调用通过 preload 暴露的限权接口。

Agent 后端子进程独立运行，即使崩溃也不会导致 UI 卡死。

6.2 文件访问控制
工具执行时校验路径是否在项目工作区内 (或用户明确允许的目录)。

禁止访问系统敏感目录 (如 /etc, C:\Windows)。

6.3 Shell 命令限制
默认禁用危险命令 (如 rm -rf, format)。

支持白名单模式 (用户可配置允许的命令前缀)。

6.4 证书与 MITM
动态生成的根证书仅用于本地代理，不对外暴露。

提供一键安装/卸载证书功能。

1. 为什么纯JS在这个项目里完全可行？
AI生成更靠谱：AI生成JS代码的幻觉率更低，因为不需要推理复杂的泛型约束。它只需关注“在if里做什么”，而不是“怎么让这个interface匹配那个type”。

不用TS不代表代码质量会下降，你可以用更轻量的方式补上：
